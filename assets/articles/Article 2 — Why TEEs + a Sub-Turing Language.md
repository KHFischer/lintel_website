# Why TEEs + a Sub-Turing Language (revised)

## A TEE attests the binary, not the behaviour

A Trusted Execution Environment (TEE) does one cryptographic job: it produces a CPU-signed statement that a specific code image ran on specific input inside a memory region the operating system and hypervisor cannot read or modify. Concretely, on Intel SGX the loaded enclave is hashed into the `MRENCLAVE` register at launch; on AMD SEV-SNP, the measured launch produces a launch digest over the initial VM image. In both cases the hardware root of trust signs that measurement with a key rooted in the platform's attestation keys, as set out in the Confidential Computing Consortium's [Technical Analysis of Confidential Computing](https://confidentialcomputing.io/wp-content/uploads/sites/10/2023/03/CCC-A-Technical-Analysis-of-Confidential-Computing-v1.3_unlocked.pdf).

Remote attestation — the step that matters to an external verifier — is a signed quote over that measurement together with a small user-data field. The user-data field is how the runtime binds an ephemeral key (the one under which it will return outputs) to the specific binary just measured, per the attestation role terminology in [IETF RFC 9334 (RATS Architecture)](https://www.rfc-editor.org/rfc/rfc9334). The semantic content of the resulting signed statement is narrow and precise: this binary, on this input, produced this commitment. It says nothing about what the binary means, what it is supposed to compute, or whether the computation corresponds to any written specification. This is a structural feature of the protocol: the measurement is a cryptographic digest of the loaded image bytes, and the attestation schema itself contains no field for program intent or specification reference. The evidence the TEE produces is about a target environment, not a target behaviour, as framed by [RFC 9334's evidence-scope discussion](https://www.rfc-editor.org/rfc/rfc9334).

The distinction the rest of this piece depends on is this: the attestation target is a code measurement, not a specification, and the binding chain from measurement to claim is laid out in the [CVP paper (SSRN 6090766)](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6090766). Call this **the Semantic Attestation Gap**: the distance between the binary-identity claim a TEE can sign and the behavioural-identity claim a verifier actually needs. A TEE proves a program ran faithfully; it does not prove the program was the right program.

The strongest counter-case is reproducible builds plus review: if a binary is deterministically built from reviewed source, and the review established behavioural intent, the binary's hash inherits behavioural meaning through that audit trail. SLSA provenance frameworks and the reproducible-builds project push exactly this direction — see the [SLSA v1.0 Provenance specification](https://slsa.dev/spec/v1.0/provenance) and the [Reproducible Builds project documentation](https://reproducible-builds.org/docs/). The counter is insufficient, not wrong. Review establishes intent once, for one reviewer, at one point in time. Every downstream build produces a new hash whose behavioural meaning requires another review, and the remote verifier has no on-line oracle mapping new hashes to old behavioural claims. The bridge from "this binary ran" to "this intended behaviour ran" is supplied by human process outside the attestation, not by the attestation itself — and that bridge must be supplied, if at all, by whatever sits inside the measured binary.

## A general-purpose language voids the attestation's semantic content

When the measured binary is a general-purpose-language program, two forces decouple the claim from the computation. Semantics-preserving changes to source or toolchain alter the measured bytes while the program's behaviour stands still, and semantic equivalence of Turing-complete programs is undecidable in general, so no verifier procedure can decide whether two differently-measured binaries compute the same function — a direct consequence of [Rice (1953), "Classes of Recursively Enumerable Sets and Their Decision Problems"](https://www.ams.org/journals/tran/1953-074-02/S0002-9947-1953-0053041-6/S0002-9947-1953-0053041-6.pdf). The attestation binds to an artefact whose computational identity is unstable under normal software evolution, and the claim and the computation decouple over time.

A program written in Python, Rust, or C++ can express any computable function. Turing-completeness is the property at stake: the halting problem is undecidable, meaning no procedure can determine in general whether such a program terminates on a given input, established in [Turing (1936), "On Computable Numbers, with an Application to the Entscheidungsproblem"](https://www.cs.virginia.edu/~robins/Turing_Paper_1936.pdf) and treated pedagogically in Sipser, _Introduction to the Theory of Computation_. The runtime has no structural bound on what it will do. A TEE happily attests such a program; it just attests what the program is, not what the program does. That is one failure mode — undecidability of runtime behaviour.

The second failure mode is separate: measurement drift. The binary's hash is not the specification's hash, and standard build operations change the first without touching the second. Illustrating with well-documented build-drift patterns from the [Reproducible Builds documentation on build-environment variations](https://reproducible-builds.org/docs/env-variations/): a developer refactors a helper function — behaviour identical, hash changes. A compiler upgrade from Rust 1.82 to 1.84 rearranges instruction selection — identical behaviour, different hash. A dependency bump in a maths library changes a tree layout — identical behaviour, different hash. An `-O2` to `-O3` flag change — identical behaviour, different hash. The measured artefact drifts while the strategy it encodes stands still.

Name this concretely: **attestation target drift**. Same strategy, same outputs, different `MRENCLAVE`. An allocator's verification pipeline that was told "check that the code image matches this hash" breaks on legitimate maintenance. Every routine software change invalidates the reference hash an allocator was asked to trust. The verifier's choice collapses to re-admitting new hashes by hand — which reduces cryptographic verification to trust-the-vendor — or rejecting all changes, which freezes the manager's stack.

Here is the concession, stated in its strongest form. Python-in-SGX, Rust-in-TDX, and similar general-purpose stacks are not broken — they are production-grade engineering patterns producing correct attestations for the claim they make: this binary, on this input, produced this commitment. For many confidential-computing use cases — private data analytics over encrypted inputs, key escrow, confidential inference on proprietary models — that claim is exactly what is needed. For strategy verification it is not. An allocator does not want to know that a specific binary ran; the allocator wants to know that a specific strategy ran. The gap between those claims is what the rest of this piece closes, per the Level-5 general-purpose-language analysis in the [OVL paper (SSRN 6388459)](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6388459).

## Sub-Turing-completeness yields termination, determinism, bounded resources, and auditability by construction

A finite acyclic dataflow graph of total functions over typed domains guarantees the four properties adversarial verification requires — not by runtime check, but by language structure.

By sub-Turing-complete, I mean a computational model strictly less expressive than a Turing machine. The absence is structural, not policy: unbounded iteration, general recursion, and arbitrary feedback are not disallowed at runtime and caught by a monitor; they cannot be written down in the language at all, because the syntactic constructs are missing. Less expressive power, in exchange for properties the verifier can prove without running the program.

The Open Veracity Language (OVL), the specification language underlying our work, sits at Level 2 of the expressiveness hierarchy set out in Figure 1 of the [OVL paper (SSRN 6388459)](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6388459) — strictly below synchronous dataflow (Level 3) and total functional programming (Level 4), strictly above fixed-function configuration files (Level 1). A Level-2 OVL program is a finite directed acyclic graph of typed nodes, where every primitive is a total function over its declared input domain.

Four properties follow from that structure. Termination follows from acyclicity: a finite topological sort terminates in finitely many steps. Determinism follows from totality: every primitive returns on every input in its declared domain, and the composition of total functions is a total function. Bounded resources follow from the graph shape: working memory is bounded by the sum of output sizes across nodes, which is computable from the specification without running the program, per the static resource-bounds treatment in the [OVL paper (SSRN 6388459)](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6388459). Auditability is the one property that lives in a human workflow rather than a compile-time proof, so it is worth stating concretely. A reviewer holding the OVL document and its canonical hash can inspect every primitive invocation and its declared types, trace the dataflow from each named input to each named output along the DAG, and re-compute the hash by canonicalising the document — with no access to the execution environment, the manager's inputs, or any runtime trace needed to decide whether the strategy does what the document says.

Three independent communities, each facing a setting in which adversarial execution is the default assumption, converged on this design stratum. SCADE/Lustre is the synchronous dataflow language used for DO-178C Design Assurance Level A avionics software, behind the A380 flight control system and equivalents — see [Colaço, Pagano, and Pouzet (2005), "A Conservative Extension of Synchronous Data-flow with State Machines"](https://www.di.ens.fr/~pouzet/bib/emsoft05b.pdf) and the A380 flight-control deployment documented in [this HAL thesis on safety-critical qualification](https://theses.hal.science/tel-04609995v1/document). Simplicity is a sub-Turing smart-contract language with formal denotational semantics in Coq and a Merklized abstract syntax tree as content-addressed identity, introduced in [O'Connor (2017), "Simplicity: A New Language for Blockchains"](https://dl.acm.org/doi/pdf/10.1145/3139337.3139340). OVL is the analogous stratum for financial computation verification. The empirical grounding for finance is that Kakushadze's canonical alpha library constructs its factor catalogue from a small, bounded set of primitive operators — arithmetic, ranking, moving-average, correlation, and related families — well within sub-Turing expressive range, as catalogued in [Kakushadze (2016), "101 Formulaic Alphas" (arXiv:1601.00991)](https://arxiv.org/abs/1601.00991). Strategy expression does not need Turing power. Convergence across three unrelated fields under the same constraint is evidence this is a structural choice, not an idiosyncratic one.

The strongest objection to the premise is that real trading strategies carry state, recurrence, and stochasticity — rolling volatility estimates, Kalman filters, reinforcement-learning position sizers, Monte Carlo resampling — and a language that excludes these excludes what managers actually deploy. The objection has teeth. I answer its two halves on different grounds. Where state is bounded and deterministic — a rolling window, an exponential moving average, a lag operator — it unrolls into a finite acyclic graph whose depth is set by the window length, and the language expresses it natively, per the bounded-lag and windowing constructs in the [OVL paper (SSRN 6388459)](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6388459). Where state or randomness is unbounded or adversarially ingested — live model retraining inside the execution path, unseeded Monte Carlo, reinforcement-learning weight updates on the production stream — the language excludes it. Section 5 returns to why that cost is worth paying.

## The canonical hash of an OVL document is a semantically meaningful attestation target

When the measured thing inside the TEE is a canonical hash of a sub-Turing specification, the attestation binds a target whose identity is stable across implementations, compilers, and refactors — because two documents with the same canonical hash have the same behaviour by construction.

The mechanism is a canonical serialisation pipeline. An OVL strategy is authored as a YAML source document. The compiler parses it into a canonical abstract syntax tree — normalised ordering, resolved references, explicit types. That AST is serialised using [IETF RFC 8785, the JSON Canonicalization Scheme (JCS)](https://www.rfc-editor.org/rfc/rfc8785), which specifies a deterministic byte sequence for any JSON-representable value. The canonical byte sequence is hashed with SHA-256. The resulting digest is the document's canonical hash. Two authors writing the same strategy in different YAML layouts produce the same canonical hash; any change in logic produces a different one, per the canonical serialisation and content-addressed identity construction in the [OVL paper (SSRN 6388459)](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6388459).

Because the language is sub-Turing, the canonical hash does more than identify a document — it commits the verifier to behaviour, under three preconditions the system freezes explicitly. Every primitive's semantics must be fixed as a versioned specification tied to a specific primitive-library release. The canonicalisation rules themselves must be fixed: node ordering, reference resolution, type representation. The admissible primitive set at a given OVL version must be closed. With those preconditions in place, two OVL documents with the same canonical hash compute the same function from inputs to outputs, per the semantic-preservation argument in the [OVL paper (SSRN 6388459)](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6388459). Turing-complete languages admit no analogous normal form, because their non-trivial semantic properties are undecidable (see Section 2).

The two layers compose because they are orthogonal: the hardware enforces the execution boundary, the language enforces the behavioural envelope, and neither layer knows about the other. Composed, they produce the claim an allocator actually needs.

Inside the Cryptographic Verification Protocol (CVP), the binding works like this. The compiled OVL specification receives a content-addressed identifier — the `CVPID`, which is the canonical hash packaged as a stable identifier. At execution time, the runtime writes that identifier into the small free-form field the TEE includes inside its signed quote (the `REPORTDATA` field, in SGX vocabulary), and the attestation covers the binding, as specified in §§2 and 5 of the [CVP paper (SSRN 6090766)](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6090766). An allocator holding the specification document and the attestation quote can verify end-to-end: this specification, with this canonical hash, ran inside a measured enclave on this input, and produced this signed output. The semantic content of the statement is now about the strategy, not about the binary implementing it.

The concrete procurement question this mechanism lets an allocator ask is neither "can I see your code?" (which most managers will reject) nor "can I trust your audit?" (a human-process question). It becomes: _what is the canonical hash of the strategy specification whose execution produced the track record you are showing me, and will every attestation quote you deliver bind that same hash?_ The question is answerable without the manager ever revealing the specification itself, because the hash is the answer.

## The constraint is the feature: the scope is strategy expression, not strategy research

OVL excludes exactly the patterns — unbounded recursion, general feedback loops, external interaction, randomness — that would make the attestation target unstable, and that exclusion is what makes verification tractable.

The named exclusions are specific: recursive position sizing where today's size depends on yesterday's realised P&L in a non-acyclic way; reinforcement-learning policies whose weights are updated by an inner optimisation loop during execution; live-data ingestion that brings non-attested inputs into the computation mid-flight, as enumerated in the excluded-patterns section of the [OVL paper (SSRN 6388459)](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6388459).

These are not limitations the language happens to have and aims to remove over time. They are the structural requirement of the attestation model. Map two of them to the specific properties they break. Live model retraining during execution admits a control-flow path whose outputs depend on weights updated mid-run from non-attested inputs — that collapses determinism, because the same specification on the same user inputs no longer produces the same outputs. Unbounded recurrence — a general Kalman filter, a stateful RL policy with unbounded memory — admits a dependency graph whose depth is not fixed by the specification, which collapses termination (structurally guaranteed only for finite DAGs) and the static bounded-resources proof (which assumes finite node count). Accepting the exclusions is the price paid for the canonical hash committing the verifier to behaviour.

State the commercial objection plainly: some deployed strategies sit out of scope for this model. A fund whose production strategy is a reinforcement-learning agent that updates its own policy on live data cannot compile to OVL without restructuring that agent, and for some managers the restructure is commercially unacceptable. This is a structural limitation, not a transitional one — the exclusions are what make the hash-to-behaviour binding exist in the first place.

For the strategies that do fit — and Kakushadze's catalogue suggests the fit is wider than the objection implies — the scope question has a clean answer. Research, exploration, factor discovery, backtest Monte Carlo, and model selection all stay in Python, R, or the manager's research platform. OVL is the compile target for the strategy as deployed: the thing the allocator is being asked to verify ran. Call this division **the Verification Boundary**. The [CVP paper (SSRN 6090766)](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6090766) draws it explicitly as the Research Environment on one side of the compiler and the verification artefact on the other. The research pipeline is not being displaced. The Verification Boundary is the line at which the artefact becomes cryptographically addressable.
## The architectural shape is structural, not arbitrary

Hardware attestation and a sub-Turing language are the minimal composition that produces an attestation whose target is semantically meaningful. Consider two plausible alternatives and how each fails. A TEE plus reproducible builds reduces to the case Section 1 disposed of: the binary-to-behaviour bridge lives in a review process outside the attestation, and every downstream build forces a fresh off-line review. A zero-knowledge proof over a general-purpose computation replaces one trust assumption with another of the same kind: the verifier still needs a stable specification of _what_ was proven, and a Turing-complete program does not admit one. Both alternatives leave the Semantic Attestation Gap open.

Concretely: TEE-only attestation fails when a developer refactors a helper function — the hash changes, the allocator's reference hash invalidates, and the verifier's only recourse is to re-admit new hashes on trust. Language-only verification fails when the manager is also the execution authority — the specification is correct, but nothing binds the output the manager reports to the output the specification actually produced. Neither layer alone carries the allocator's claim; both together do.

The problem this architecture is solving — the Proof-Execution Gap between what a manager claims happened computationally and what actually happened — is the subject of Article #2 in this sequence. The formal treatment of the language layer is in the [OVL paper (SSRN 6388459)](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6388459); the protocol-level binding chain between specification hash and hardware quote is in the [CVP paper (SSRN 6090766)](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6090766). An allocator reading this should ask two questions of any manager claiming cryptographic verification of their strategy. First, what is the attestation target — a binary measurement, or a canonical hash of a behaviourally stable specification? Second, will next month's attestation bind the same specification hash as this month's? The first question separates closers of the Semantic Attestation Gap from those who hide it. The second holds the closer accountable through time.
